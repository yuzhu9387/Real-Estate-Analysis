AlphaX Project Management System

A platform to manage residential development projects across their full lifecycle — from land acquisition through entitlement, permitting, construction and sales — with workflows, tasks, financial pro formas, team coordination and a Lark data integration. This document explains how every page, feature and object works.

★ Key Concepts (read first)

Five ideas underpin everything else in the system.

1. Business Lines (the 4 brands)

Every project belongs to one business line. The list is the single source of truth — every dropdown reads from it.

AL Homes SFHAL Homes Community AleraApex

2. The 5 Lifecycle Phases

A project moves sequentially through five phases. Financials live in a separate Investment tab shown alongside them.

Acquisition→ Entitlement→ Permitting→ Construction→ Sales +Investment
Phase vs Display Phase: internally the database uses a larger "FSM" vocabulary (evaluation, purchase, government, final, closeout…). These are collapsed into the 5 display phases for the UI. An ON_HOLD project shows its previous phase and is colored orange.

3. Project Hierarchy: Project → Stage → Unit

A project can be broken into child nodes. The node_level field marks each node.

Project

The root node — the overall development.

Stage

A phase-level sub-division of a project (e.g. a sub-phase or split).

Unit

An individual sellable building. SB9 lot-splits create several unit rows.

4. Workflows: Template → Project Workflow → Task

Repeatable processes are authored once as templates, then snapshotted onto a project's phase as live tasks.

Workflow Template→ assign to phase → Project Workflow→ becomes → Tasks (with owners, dates, dependencies, reviews)

5. Roles control what you can do

Three global roles plus per-project role assignments and per-table permission grants.

Teams (functional departments)

A Team is a departments row flagged is_team = true. Teams own workflows (a workflow template/instance has an owner_department_id) and appear on the Teams board, the milestone color legend and the project-role pickers. The 9 default teams, in order:

#TeamOwns / typically responsible for
1AcquisitionLand sourcing, offers, due diligence, closing
2ArchArchitectural design & entitlement/building packages
3CivilCivil engineering & jurisdiction approvals
4Interior DesignInterior design (ID/ID-CD packages, material lists)
5UtilityUtility coordination
6Exterior PackageExterior & landscape package
7ConstructionBuild execution & construction milestones
8SalesPresales, listing & closing
9Project ManagementOverall coordination across teams
Teams vs Departments vs Project Roles: Departments are the org units; those flagged is_team become assignable Teams. Project Roles (Acquisition Lead, Arch Lead…) are the per-project people assignments and largely mirror the teams.

7 Glossary

Definitions for the domain terminology used throughout the system.

General & Lifecycle

TermDefinition
Business Line (Brand)Top-level division a project belongs to: AL Homes SFH, AL Homes Community, Alera, Apex. "Brand" is an alias used by the workflow library.
Pipeline CategoryA sub-classification under a business line; can carry its own phase overrides.
Display PhaseThe phase shown in the UI: the edited override if valid, else derived from the DB phase.
FSMThe DB phase state machine — a larger legacy vocabulary collapsed into the 5 display phases.
ON_HOLDAn orthogonal state; for display it resolves via the previous phase and renders orange.
Product TypeWhat is being built (SB9 Lot Split, SB 684, Community…). 8 selectable + 16 legacy codes.
SB9 / SB 684 / SB 1123California housing statutes enabling lot splits / multi-unit infill — used as product types.
Kick OffSetting the phase start date, which locks the schedule/timeline engine.
Pipeline MatrixDashboard pivot of leaf units by exit year × business group.
Luxury threshold$7,000,000 target exit price — splits AL Homes SFH into Luxury vs Standard.

Scheduling & Workflows

TermDefinition
Planned / Target / ActualDay offsets from kick-off (planned), the calendar plan (target), and recorded dates (actual).
Phase cascadeSequential schedule derived from start date + per-phase durations.
Critical PathTasks flagged as driving the overall timeline.
Dependency (finish-to-start)Task B starts after Task A finishes + lag days. The only dependency type.
Slip / Phase deltaLateness: a phase's actual end vs the next phase's planned start.
Workflow TemplateA reusable task sequence with dependencies, snapshotted onto projects.
Unplanned / Blocked taskAdded outside the template / user-flagged as blocked.
Review flowOwner submits → reviewer approves or requests revision.

Finance / Proforma

TermDefinition
ProformaA versioned financial model of a project: cost/revenue inputs → computed returns.
Line itemOne cost/revenue row, computed via per_sf, pct_of_base, per_year or manual.
Hard / Soft / ContingencyCost types: physical construction / design-permits-legal / reserve.
Holding costsCarrying costs (property tax, insurance, HOA) spread over the holding period.
Cost curveHow a cost spreads across months: bell, steady_growth, steady_decrease.
Gross ProfitNet sale revenue − total cost.
Net Profit (levered)Unlevered net profit − loan fees − interest.
IRRInternal Rate of Return (levered & unlevered).
EM (Equity Multiple)Total inflows ÷ total outflows.
ROIUnlevered net profit ÷ pre-interest cost.
Loan termsAcquisition & construction loans (rate, amount) plus points (origination fee %).
Sale economicsSale price, flat listing fee, selling commission %, closing/title %.
LedgerActual posted spend/revenue per phase/category, rolled up against the plan.

Permits & Integration

TermDefinition
Permit StageA named permit-workflow step scoped to a business line + pipeline category.
Stage flowHow a stage runs: Milestone (single date), Parallel (together), Sequential (ordered).
Lark (Feishu)Collaboration platform integrated for Bitable/Sheet/Doc import & contact sync.
BitableLark's structured multi-dimensional table; a primary import source.
Field mappingA rule mapping a source column → a target DB field, with an optional transform.
Full vs Delta syncFull re-import vs only-changed-rows import.
StagingImported rows held for review (create/update/skip; approve/reject) before commit.
